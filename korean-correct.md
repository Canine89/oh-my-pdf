맞춤법 검사 루틴 명세

  이 맞춤법 검사 시스템은 한국어 기술서, 교재, 출판 원고를 대상으로 문장을 교정한다. 단순 맞춤법뿐 아니
  라 출판사 내부 용어표, GPT 기반 문체 검토, 과교정 방지 필터, PDF/문서 처리용 결과 구조화를 함께 사용
  한다.

  입력은 일반 텍스트, Google Docs 문단, PDF에서 추출한 페이지 텍스트 중 하나다. 처리 대상이 어디서 왔는
  지는 다르지만, 실제 교정 로직은 동일한 파이프라인을 따른다.

  입력값

  검사 대상 텍스트와 함께 다음 옵션을 받는다.

  {
    "text": "검사할 문장 또는 문단",
    "use_gpt": true,
    "gpt_model": "gpt-4.1-mini",
    "correction_strength": "balanced",
    "use_publisher_rules": true,
    "join_auxiliary_verbs": true,
    "check_passive_voice": false,
    "tone": ""
  }

  옵션 의미는 다음과 같다.

  use_gpt: GPT 교정 사용 여부
  gpt_model: 사용할 GPT 모델. 기본값은 gpt-4.1-mini
  correction_strength: 과교정 감지 강도
  use_publisher_rules: 출판사 용어표 적용 여부
  join_auxiliary_verbs: 보조용언 붙여쓰기 적용 여부
  check_passive_voice: 피동 표현 교정 여부
  tone: 문체 옵션. 빈 값이면 원문 유지, textbook이면 수험서체로 변환

  교정 강도

  교정 강도는 GPT 결과를 얼마나 엄격하게 받아들일지 결정한다. 내부적으로는 원문과 교정문 사이의 유사도
  임계값으로 사용한다.

  ultra_minimal: 0.98
  minimal: 0.90
  conservative: 0.75
  balanced: 0.65
  aggressive: 0.45
  very_aggressive: 0.25

  값이 높을수록 원문과 많이 달라진 교정을 버린다. 즉 ultra_minimal은 거의 확실한 교정만 허용하고,
  very_aggressive는 더 많은 문체 변경을 허용한다.

  전체 처리 순서

  1. 입력 텍스트가 비어 있으면 그대로 반환한다.
  2. 입력이 명백한 코드 블록이면 교정하지 않고 그대로 반환한다.
  3. 출판사 용어표 규칙을 먼저 적용한다.
  4. GPT 사용 옵션이 켜져 있으면 GPT로 추가 검토한다.
  5. GPT 응답을 JSON으로 파싱한다.
  6. GPT 결과가 과교정이면 GPT 결과를 버리고 이전 결과를 유지한다.
  7. GPT가 제안한 교정 항목 중 잘못된 항목을 필터링한다.
  8. 출판사 규칙 교정과 GPT 교정을 합성한다.
  9. 보조용언 결합 옵션이 켜져 있으면 보조용언 스타일을 적용한다.
  10. 최종 교정문과 교정 항목 목록을 반환한다.

  출판사 용어표 적용

  출판사 용어표는 GPT보다 먼저 적용한다. 용어표는 여러 시트 또는 카테고리로 구성되며, 정해진 순서대로
  적용한다.

  예를 들어 다음과 같은 규칙이 있을 수 있다.

  [
    { "original": "깃 헙", "corrected": "깃허브" },
    { "original": "할수 있다", "corrected": "할 수 있다" },
    { "original": "되어 보", "corrected": "되어보" }
  ]

  일반 치환 규칙은 단순 문자열 치환한다. 다만 공백을 제거하거나 추가하는 규칙은 단어 중간에서 잘못 매칭
  되지 않도록 어절 경계를 확인한다.

  예를 들어 "일 부" → "일부" 규칙이 있을 때 "이메일 부분만" 안의 "일 부"처럼 보이는 문자열은 교정하지
  않는다.

  GPT 교정

  GPT는 출판사 용어표가 적용된 텍스트를 다시 검토한다. GPT에는 다음 원칙을 강하게 지시한다.

  - 맞춤법, 띄어쓰기, 조사, 번역투, 문체 오류만 수정한다.
  - 의미를 바꾸지 않는다.
  - 문장 구조를 바꾸지 않는다.
  - 원문에 없는 단어를 추가하지 않는다.
  - 코드 문단은 수정하지 않는다.
  - 고유명사, 브랜드명은 임의로 바꾸지 않는다.
  - 수정할 내용이 없으면 설명하지 말고 원문을 그대로 반환한다.

  GPT 응답은 다음 JSON 형식을 기대한다.

  {
    "corrected_text": "교정된 전체 텍스트",
    "corrections": [
      {
        "original": "원문에서 실제로 존재하는 단어 또는 구문",
        "corrected": "수정된 단어 또는 구문",
        "type": "spelling",
        "explanation": "교정 이유"
      }
    ]
  }

  type은 주로 다음 값을 사용한다.

  spelling: 맞춤법, 띄어쓰기, 문법
  style: 문체, 번역투 표현
  publisher_rule: 출판사 용어표 규칙

  GPT 응답 검증

  GPT 응답은 그대로 믿지 않는다. 다음 조건에 해당하면 원문 또는 이전 단계 결과를 유지한다.

  - JSON 파싱에 실패한 경우
  - “수정할 내용이 없습니다”, “맞춤법이 올바릅니다” 같은 설명형 응답을 한 경우
  - 교정 결과가 원문과 지나치게 달라진 경우
  - 교정 항목의 original이 실제 원문에 존재하지 않는 경우
  - 원문에 없던 내용을 추가한 것으로 보이는 경우
  - 고유명사나 보호 단어를 훼손한 경우
  - 줄바꿈, 개행, 주어 보완, 누락 보완 같은 설명을 근거로 내용을 추가한 경우
  - 공백만 줄여서 붙이는 식의 위험한 교정인 경우

  즉 이 시스템은 GPT를 “최종 판단자”로 쓰지 않고, “제안자”로 쓴 뒤 후처리 필터를 통과한 결과만 반영한
  다.

  과교정 감지

  GPT 교정 후 원문과 교정문의 유사도를 비교한다. 유사도가 설정된 임계값보다 낮으면 과교정으로 판단하고
  GPT 결과 전체를 버린다.

  예를 들어 balanced 모드에서는 임계값이 0.65다. 교정문이 원문과 너무 많이 달라지면 문체 개선이 아니라
  의미 변경일 가능성이 높다고 보고 폐기한다.

  출판사 규칙과 GPT 결과 합성

  출판사 규칙과 GPT가 같은 부분을 연속으로 바꿀 수 있다.

  예:

  원문 A
  출판사 규칙: A → B
  GPT 교정: B → C
  최종 표기: A → C

  이 경우 결과에는 A → B, B → C를 따로 표시하지 않고 A → C로 합성한다.

  반대로 GPT가 출판사 규칙을 되돌리는 경우:

  원문 A
  출판사 규칙: A → B
  GPT 교정: B → A

  최종 변경 효과가 없으므로 두 교정을 모두 제거한다.

  보조용언 처리

  join_auxiliary_verbs 옵션이 켜져 있으면 최종 교정문에 보조용언 결합 규칙을 적용한다. 예를 들어 출판사
  스타일상 붙여 쓰는 표현을 통일하기 위한 단계다.

  이 단계에서 이미 교정 항목으로 잡힌 값이 보조용언 결합 후 의미 없는 항목이 되면 결과 목록에서 제거한
  다.

  코드 블록 제외

  다음과 같은 텍스트는 코드로 판단하고 교정하지 않는다.

  - def, class, import, return 등으로 시작하는 코드
  - const, let, var 선언
  - SQL 문
  - 함수 호출 또는 중괄호 기반 코드
  - 한국어가 없고 세미콜론, 연산자, 중괄호가 많은 텍스트

  목적은 코드 예제 안의 변수명, 함수명, 명령어를 GPT가 한국어처럼 교정하는 것을 막는 것이다.

  PDF 처리 방식

  PDF는 파일 전체를 한 번에 GPT에 넣지 않는다.

  1. PDF를 업로드한다.
  2. 총 페이지 수를 확인한다.
  3. 페이지별 텍스트를 추출한다.
  4. 각 페이지 텍스트를 맞춤법 검사 파이프라인에 넣는다.
  5. 페이지별 결과를 저장한다.
  6. 교정 항목이 있는 페이지를 집계한다.
  7. 원본 PDF 위에 교정 위치를 하이라이트한 PDF를 생성한다.

  페이지 결과는 다음 구조를 가진다.

  {
    "page_number": 3,
    "original_text": "원문 페이지 텍스트",
    "corrected_text": "교정된 페이지 텍스트",
    "corrections": [
      {
        "original": "됬습니다",
        "corrected": "됐습니다",
        "type": "spelling",
        "explanation": "맞춤법 교정"
      }
    ],
    "has_errors": true
  }

  PDF 검사는 페이지 단위 병렬 처리를 사용한다. 기본적으로 여러 페이지를 동시에 GPT 검사할 수 있다.

  Google Docs 처리 방식

  Google Docs는 문서 URL을 받아 문단 단위로 처리한다.

  1. Google Docs URL에서 문서 ID를 추출한다.
  2. 문서의 문단을 가져온다.
  3. 각 문단을 검사한다.
  4. 코드 문단이나 형식 문단은 제외한다.
  5. 수정이 필요한 문단을 교정한다.
  6. 결과를 저장한다.
  7. 옵션에 따라 Google Docs에 직접 반영한다.
  8. 진행률을 큐 작업 상태에 저장한다.

  작업 상태는 다음 중 하나다.

  pending: 대기 중
  processing: 처리 중
  completed: 완료
  failed: 실패
  cancelled: 취소됨

  최종 응답 형식

  일반 텍스트 검사 결과는 다음 구조를 권장한다.

  {
    "original": "원문",
    "corrected": "교정문",
    "diff_html": "변경 부분을 표시한 HTML",
    "use_gpt": true,
    "gpt_model": "gpt-4.1-mini",
    "corrections": [
      {
        "original": "할수",
        "corrected": "할 수",
        "type": "spelling",
        "explanation": "띄어쓰기 교정"
      }
    ]
  }

  핵심 설계 원칙

  이 시스템의 핵심은 다음이다.

  1. 출판사 용어표를 GPT보다 우선한다.
  2. GPT는 교정 제안만 하고, 후처리 필터가 최종 반영 여부를 결정한다.
  3. 원문에 없는 단어를 추가하는 교정은 금지한다.
  4. 의미 변경보다 최소 편집을 우선한다.
  5. 코드와 고유명사는 최대한 보호한다.
  6. PDF와 Google Docs는 입력/출력 방식만 다르고, 교정 엔진은 같은 로직을 사용한다.
  7. 장시간 작업은 큐 또는 백그라운드 작업으로 처리한다.
